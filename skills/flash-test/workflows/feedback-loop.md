# 反馈闭环（被动触发）

**触发**：用户说「换了个 UFS」「再看结果」「问题解决了」「修复生效」「重新测一下对比」等。

不在主流程自动弹出，仅用户主动触发时读取本文件。

## 动作 1：对比历史

读 `data/patterns.json`，对比本次与同型号历史 `test_results`：
- **退化**（数值下降）→ 排查调度器 / WB 状态 / 温度 / 后台负载变化。
- **提升**（数值上升）→ 确认环境变化（如开了 Write Booster、换了内核/调度器）。

## 动作 2：确认规格来源

若换了芯片，重新走规格匹配：
- 检测新型号 → 查 `device_profiles.<model>.spec`；未收录则询问用户提供 → 更新 profile。

## 动作 3：更新置信度

对 `alerts` 中本次复现 / 证伪的异常条目，调整 confidence（复现 +1，证伪 -1，<0 移除）。这是 skill「越用越完善」的核心机制。

## 动作 4：一句话总结

回应用户：本次 vs 上次 + 规格达标情况 + 关键变化。

> 反馈闭环默认不重跑测试（除非用户明确要求）；其价值在横向对比与规格/置信度维护。
