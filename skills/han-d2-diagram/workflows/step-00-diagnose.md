# Step 0: 诊断用户需求

快速判断用户意图，决定工作流入口点。

## 诊断流程

### 1. 检查用户输入类型

- **明确图表类型**："画个流程图"、"画个ER图"、"画个类图" → **快速创建模式**
- **抽象需求**："把这段架构描述转成图表"、"根据需求画图" → **需求理解模式**
- **已有代码**："帮我改下这个 D2"、"优化下布局" → **代码优化模式**
- **语法查询**："D2 怎么画类图"、"shape: class 怎么用" → **参考查询模式**

### 2. 模式处理

#### 快速创建模式
- 直接询问用户具体需求细节（节点、关系、样式）
- 跳过详细需求分析，直接进入 Step 1 选择对应模板
- 可直接使用 `assets/templates/<type>.d2` 作为起点

#### 需求理解模式
执行深度需求分析：

1. **提取实体** — 从用户描述中识别核心对象、属性/字段、数量估算
2. **提取关系** — 识别连接类型：数据流(A→B)、依赖关系、包含关系、通信关系
3. **推断图表类型**：
   - 有决策分支？→ 流程图
   - 有表结构/字段？→ ER 图或类图
   - 有时间线/阶段？→ 甘特图
   - 有状态转换？→ 状态机
   - 有时序交互？→ 序列图
   - 是系统分层？→ 架构图
4. **确认样式偏好**：
   - 技术文档？→ `--theme=300`
   - 演示文稿？→ `--sketch`
   - 暗色环境？→ SVG + `--dark-theme=200`

#### 代码优化模式
- 跳过 Step 0-1，直接进入 Step 2
- 如果只是样式问题，直接进入 Step 3

#### 参考查询模式
- 不进入工作流，直接查阅 `references/` 返回答案

### 3. 决策树

```
用户输入
├─ 明确图表类型？ → 是 → 快速创建 → Step 1
├─ 已有 D2 代码？ → 是 → 代码优化 → Step 2 或 Step 3
├─ 只是语法查询？ → 是 → 查阅 references/ → 结束
└─ 抽象需求 → 需求理解 → 深度分析 → Step 1
```

## 完成标志

- 确定了用户模式
- 需求理解模式下已提取实体+关系+图表类型+样式偏好
- 明确下一步入口
