# /llm-wiki think —— 10 原则深度思考

应用 10 原则思维循环，对复杂问题进行结构化深度思考。每个原则代表思考的一个阶段，按顺序走过，形成从观察到创造的完整循环。

**触发条件：**
- `/llm-wiki think <问题>`
- `/llm-wiki think` — 对当前对话中悬而未决的问题进行思考
- 用户说「深度思考」「10 原则审视」「想清楚」「系统思考一下」

**原理：** 10 原则不是一份检查清单，而是注意力的阶段导航。你按顺序走第一遍，当新信息出现时循环回早期阶段。关键是**不跳过不舒服的阶段**（内部观察、接受、成长）。

---

## 10 原则（中文适配版）

### 1. 观察（外部输入）

**思考从数据收集开始。** 看原始输入、代码、日志、用户描述——不急于下结论。

- 在行动前：读代码再改它。读日志再下判断。读上下文再回答。
- 禁用词：「肯定是...」「这简单，就是...」（说明跳过了观察）

### 2. 观察（内部认知）

**审视自己的思考方式。** 我现在有什么假设？对这个问题有偏见吗？是否锚定在之前的经验上？

- 写下 1-2 句「当前偏见」——承认它不等于消除它，等于把它的影响控制在可观测范围内
- 常见偏见：熟悉感偏见（以前见过类似的）、收工偏见（想快点结束）、锚定（第一个方案占据思维）

### 3. 倾听（主动接收）

**观察偏分析，倾听需要放下自我。** 理解用户真实意图、错误信息的深层原因、沉默里的需求。

- 读用户的**原话**而不是你的转述
- 读**失败信息**而不是猜测失败原因
- 用户的困惑本身就是数据

### 4. 思考（批判性分析）

**分析引擎启动。** 用第一性原理拆解问题。结构化推理，映射工作流，评估约束。

- 核心工具：读后写、最小可用单元、删除多于增加、证据优于直觉
- 追问「为什么」直到不能再拆

### 5. 连接（横向/联想）

**大创意很少凭空产生，都在交叉点。** 把两个看似无关的概念放在一起。

- 这个 bug 模式在别的模块存在吗？
- 这个架构问题和那个架构问题是同构的吗？
- 横向思考发现的事，逐组件审查会漏掉

### 6. 连接（系统/编排）

**从孤立想法走向集成系统。** 各个组件如何连接成一个运转的整体？

- 新功能会影响 hooks/传输/锁/路由吗？
- 一个部件的改动会不会让另一个部件出问题？
- 系统思考关注的是「彼此之间的关系」

### 7. 感受（情商 + 直觉）

**纯逻辑没有同理心会变脆。** 错误信息的用户体验好吗？用户的第一感受是什么？

- 一个错误码 "ERR: exit code 4" 不如 "DMA 传输失败：目标缓冲区已释放"
- 信任你的经验直觉，当数据模糊时它是重要的决策依据

### 8. 接受（理性谦卑）

**没有计划能在现实中存活。** 拥抱约束，承认失败，放弃沉没成本。

- 诚实地分等级：78/100 就是 78/100，不要写成 95/100
- 如果结论不是你要的，不要改结论——改假设

### 9. 创造（生成输出）

**分析瘫痪是进步之敌。** 在某个时刻，停止策略思考，开始输出。

- 一个 B+ 的文档比一个永远没写完的 A+ 文档好
- 一个能用的 v1 比一个永远在设计的 v2 好
- 前面的阶段给了你足够的信息——现在行动

### 10. 成长（迭代循环）

**思考不是直线，是反馈循环。** 从创造中观察结果，升级技能，投入下一轮。

- 这次思考有什么可以改进的？
- 什么影响了下一次？
- 成长是把一个好决策变成复利习惯的东西

## 工作流

1. **接收问题**：用户提出问题或决策。如果无参数，回顾当前对话找出最值得深思的开放问题
2. **走过 10 阶段**：按顺序输出每个阶段的思考（每阶段 2-4 句，不要展开成论文）。判断力：简单的阶段一笔带过，关键阶段深入
3. **综合结论**：基于 10 阶段的思考给出最终判断。格式：
   ```
   🎯 结论：[一句话结论]
   
   理由：[关键推理链]
   
   ⚠️ 剩余不确定性：[诚实标注]
   
   🌱 成长记录：[本次思考的方法论收获]
   ```
4. **可选归档**：如果思考结果有价值，主动建议 → 「是否将此思考保存到 wiki/analyses/？」

---

## 与 query 集成

`/llm-wiki query --think` 或 `深度回答`：在综合回答前先走一遍 think 循环，确保回答经过严格审视。所有引用都标注 wiki 来源。最终回答同时包含思考过程和结论。

---

## 使用场景

- 架构决策：「这个模块应该用中断还是轮询？」
- 技术选型：「Rust 还是 C 来实现这个驱动？」
- 疑难分析：「为什么这个 bug 修了三次都没修好？」
- 风险评估：「这个改动合入主线有哪些风险？」
- 方法论反思：「过去这个项目为什么总是延期？」
