# Step 0: 需求捕获与架构选择

在创建 skill 之前，先和用户对齐需求和架构选择。

## 0.0 前置闸门

动手前先自检，命中以下任一就别急着做 skill：

- **一次性/临时任务**（不会复用）→ 直接帮用户做掉，说明"这种用一次的，不必做成 skill"，退出本流程。
- **Claude 原生能力足够**（通用问答、简单转换）→ 坦诚告知用户直接对话即可。
- **已有类似 skill** → 先查重：`ls ~/.claude/skills/*/SKILL.md ~/.ai_utils/skills/*/SKILL.md 2>/dev/null | xargs grep -li "关键词"`，命中则建议复用或扩展（扩展走 Step 4 迁移），避免重复造轮子。
- **范围过大**（一个想法其实是好几个 skill）→ 提醒拆分，本次聚焦其中一个。

闸门通过，再进入 0.1。这是 AI 自检准则，不是提问——只在明显命中时劝退，不啰嗦。

## 0.1 捕获意图

沿用官方 skill-creator 的需求捕获方法，**一次一问、逐个确认**（给 2-3 个选项让用户挑，而非凭空逼问）。明确四个核心问题：

1. **这个 skill 要做什么？** — 核心功能
2. **什么时候触发？** — 触发信号/用户短语
3. **期望的输出格式？** — 产物形式
4. **是否需要测试用例？** — 输出是否可客观验证

### 水平自适应（不打断问"你几级"）

全程靠对话信号动态判断用户水平，调整术语密度和问题颗粒度：

| 观察到的信号 | 判断 | 对话调整 |
|---|---|---|
| 用具体文件名/工具名/数据结构（`.xlsx`、`DataFrame`、`user_id` 列） | 偏专业 | 可用术语，问题精炼直接 |
| 自己给出输入输出样例 | 偏专业 | 围绕样例确认转换规则，不必从头问 |
| "那种东西""差不多""不知道咋说" | 偏小白 | 大白话 + 类比 + 给选项让他挑 |
| 把 skill 当万能工具、想塞大量场景 | 需引导收敛 | 先聚焦一个核心能力，其余记为后续迭代 |

水平判断不是一次性的——对话推进中随时更新。

## 0.2 架构选择

用 `AskUserQuestion` 让用户选择架构（先解释"拆分+自学习"的收益）：

```
question: "请选择 skill 的架构方案？"
header: "架构选择"
options:
  - label: "文件拆分 + 自主学习（推荐）"
    description: "SKILL.md 做路由(≤100行)，workflows/ 分步骤按需加载，data/ 自学习越用越完善。省 token、好维护、能进化。适合复杂多步骤的 skill。"
  - label: "仅拆分文件，不加自主学习"
    description: "按步骤拆成多个文件，省 token 好维护。但没有 feedback-loop 和 patterns.json 自学习。适合步骤固定、不需要持续优化的 skill。"
  - label: "传统单文件（标准 skill）"
    description: "所有内容写在一个 SKILL.md 中，简单直接。建议使用官方 skill-creator（说"创建一个 skill"即可）。"
```

### 选择处理

- **文件拆分 + 自主学习**：继续 Step 1 → Step 2 → Step 3 → Step 5
- **仅拆分文件**：跳过 Step 2（自学习），执行 Step 1 → Step 3 → Step 5
- **传统单文件**：告知用户使用官方 skill-creator，本 skill 不再介入

## 0.3 确认 skill 名称和目录

询问用户目标 skill 的名称和存放位置：
- 如果在当前项目（`.ai_utils/skills/`）下创建，使用 `skills/<skill-name>/`
- 如果创建独立 skill，使用用户指定的路径

## 常见卡点破解话术

需求挖掘遇到以下卡点时，对症下药：

- **用户说不清产出是什么**（"就是帮我处理一下""让它更好"）→ 从用途倒推："如果这个 skill 做好了，你下一步拿结果做什么？" 用户对用途的描述往往比对产出更清晰，反推产出形态更有效。
- **需求太大，想把所有场景塞进一个 skill** → 帮用户排优先级而非拒绝："这几个场景里哪个最频繁、最想先解决？先做好这个，其余第一版跑通后再扩展。" 解释「一个 skill 聚焦一件事，触发更准」比直接说"太多了"更易接受。
- **例子太少或只有一个** → 追问防过拟合："这是最常见的输入，还是特殊情况？能再给一个不同的例子帮我确认规律吗？" 目的是让 skill 学到规律而非记住单个案例。

## 0.4 确认门（动手前回述）

四要素和架构都齐了，开写前必须把理解回述给用户确认：

> **需求小结**
> - 做什么：[一句话核心功能]
> - 何时触发：[触发信号]
> - 输入/输出：[输入形态 → 产物形态]
> - 边界：[不处理什么 / 何时交还人工]
> - 架构：[拆分+自学习 / 仅拆分]

问："我的理解是这样——准确吗？要补充或修正吗？没问题我就开始写了。"

**理解没对齐，绝不开写。** 这一步省掉后面 80% 的返工。

**完成后，读取 `workflows/step-01-gen-split-arch.md` 继续。**
