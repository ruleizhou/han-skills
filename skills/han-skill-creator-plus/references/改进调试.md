# 改进调试：E5 诊断剧本

用户已有 skill 但「不好用」或「不触发」时走本流程。核心逻辑：**先读懂现有 skill，定位症状，再有针对性地改**。盲目改动往往越改越乱。

## 一、入口：先读懂现有 skill

动手前通读：
- `SKILL.md`（frontmatter 的 `name`/`description` + 正文指令逻辑）
- `references/` 下所有文件（了解 skill 依赖的知识结构）
- 如有 `evals/`，扫一眼已有测试用例

读完后给选项锁定症状（别等用户自己分类）：
**「你观察到的问题是什么——是根本没被触发、触发了但输出不对、触发太频繁/太少、还是输出太啰嗦？」**

## 二、诊断决策树

症状 → 检查点 → 动作，三步走。

### 症状 A：不触发

**检查点：description 字段**
- 有没有明确说「做什么」（核心能力）和「何时用」（触发场景）？
- 是否太窄——只命中某个具体措辞，用户平时不会那么说？
- 是否太泛——和其他 skill 或 Claude 通用能力完全重叠？

**动作**：重写 description。参照 `skill-writing-guide.md` 第三章——把何时使用的信息全放进去、主动防 under-trigger、触发短语多样化。

### 症状 B：行为跑偏 / 输出不对

**检查点一：指令是否含糊**
- 输出的格式、长度、语气说清了吗？
- 有没有只堆 MUST/NEVER、不给理由？（光靠禁令，模型在边界情况容易失判）
- 有没有具体例子或参考样本？（对固定输出格式的 skill，一两个例子胜过两百字描述）

**检查点二：references 是否缺位**
- skill 依赖的领域知识（格式规范、术语、上下文）有没有放进 `references/`？没有的话模型只能靠训练知识猜，稳定性差。

**动作**：把含糊指令重写成**「目标 + 理由 + 示例」三段式**，而非加更多禁令。需要的 references 在 SKILL.md 里标注何时去读。

### 症状 C：过触发 / 欠触发

**检查点：description 边界是否清晰**
- 过触发 = description 覆盖太宽，和相邻 skill / Claude 通用能力产生模糊地带
- 欠触发（随机）= 触发短语太单一、太依赖某个关键词

**动作**：收窄 description 范围 + 增加「近似负例」（用了相似词但不该触发的情况）作为区分。注意：简单单步任务本来就不触发 skill，别强求。

### 症状 D：太重 / 太啰嗦

**检查点：指令是否过度堆砌**
- 大量「先确认 X 再检查 Y 最后才能 W」的过度细化步骤？
- 大段只在少数情况才用到的背景解释塞进了正文？
- SKILL.md 超过 500 行？几乎肯定有可下沉的内容。

**动作**：精简 + 分层。删掉「不拉车的指令」（删了也不影响核心输出的）；把只在特定场景才用的细节移到 `references/`，正文留一行「如遇 X 情况，读 `references/xxx.md`」。SKILL.md 该是路线图，不是操作手册。

## 三、改完即验证

每轮改动后别靠直觉判断「应该好了」：

- **默认轻量**：构造 2–3 个代表性 prompt，直接跑一遍给用户看输出。
- **重型量化按需**：要严格对比「改完比之前好多少」时，走官方 skill-creator 的评测流程（baseline + benchmark + eval viewer）。
- **description 改动专项**：若改了 description，额外确认触发准确率有提升。

## 四、踩坑提醒

- **保留 name 和目录名不变**。skill 身份由目录名 + frontmatter `name` 共同决定，改名会导致安装路径变、旧引用断、肌肉记忆失效。无论改多大，`name` 保持原样。
- **只读路径先拷贝再编辑**。已安装的 skill 路径可能只读，直接改报权限错误。先 `cp -r <skill-path> /tmp/<skill-name>/`，在副本上改，打包后覆盖回去。

## 五、改进建议分类框架

诊断定位问题后，「从哪改」用这 6 个维度归类，按优先级排序动手：

| 类别 | 改什么 | 举例 |
|------|--------|------|
| `instructions` | 正文指令 | 把「适当处理」改成明确的 3 步 |
| `tools` | 脚本/工具 | 补一个 `validate_output.py` |
| `examples` | 示例 | 加一对输入→输出样例 |
| `error_handling` | 失败处理 | OCR 失败的 fallback 步骤 |
| `structure` | 内容组织 | 步骤拆分或合并 |
| `references` | 外部文档 | 补领域知识文件 |

**优先级判定**：
- **high**：会改变本次结果（改了 outcome 就不同）——优先做
- **medium**：提升质量但不一定改变胜负
- **low**：锦上添花

判定标准：这条改动**会不会改变最终输出**？会→high。每条建议附一句 `expected_impact` 说清预期效果，别只说「改这里」。
