# Skill 写作指南

生成或改写 SKILL.md / 步骤文件时遵循本指南。源自 skill-creator-cn 写作实践，中文化适配。

## 一、写作风格准则

写 skill 正文（步骤指令、规则说明）时：

- **用祈使句写指令**。「读取文件并校验格式」比「你应该读取文件然后校验格式」更容易被执行。
- **解释「为什么」，而非堆 MUST/NEVER**。今天的 LLM 足够聪明，给出理由它能举一反三；光靠全大写禁令，模型在边界情况下反而失去判断力。**当你发现自己写下 ALWAYS/NEVER 这类全大写词——那是警示信号**，试着改写成解释理由的方式。
  - 反例：`NEVER 在报告里输出原始 token 数`
  - 正例：`不要在最终报告里暴露 token 数——用户只关心结论，原始数值会干扰阅读`
- **写完换个视角重读一遍**。这步常能发现自己没意识到的问题：指令是否会被理解成两种意思、有没有遗漏前提。
- **防过拟合**。你在做的是「能被用 100 万次的 skill」，现在只是在几个例子上快速迭代。若改出来的规则只对当前例子有效，就失去了意义。挖规律，别记案例。

## 二、草稿自检四件事

SKILL.md 和各步骤文件**生成后**，先自检再交给用户，重点扫：

| 问题 | 表现 | 处理 |
|------|------|------|
| **占位含糊** | "TBD""按需处理""适当地……"这类没说清的要求 | 说清它，或删掉 |
| **自相矛盾** | 前后指令打架、description 与正文不一致 | 统一成一种 |
| **范围蔓延** | 混进了需求确认时没覆盖的功能 | 砍掉，或回去和用户确认 |
| **歧义** | 某条要求能被理解成两种意思 | 挑一种、写明确 |

发现问题就地改掉，不必反复评审。

## 三、description 写作指南

`description` 是 Claude 决定是否调用 skill 的**核心触发器**——它决定 skill 会不会被用。

**触发机制本质**：Claude 通过 `name + description` 判断是否调用，且只在自己无法轻松处理的复杂任务上才会咨询 skill。简单的单步操作（如「读这个 PDF」）即使 description 很匹配也可能不触发——Claude 直接用基础工具搞定，这是正常行为，不是 bug。

**写法**：
- **把所有「何时使用」的信息都放进 description，不要藏在正文里**。
- **主动一点，防 under-trigger**。Claude 常常「明明适合用却没用」，所以描述要稍微「推销自己」。
  - 反例：`如何构建仪表盘`
  - 正例：`构建仪表盘。当用户提到仪表盘、数据可视化、内部指标、想展示任何公司数据时，即使没有明确说"仪表盘"，也应使用本 skill。`
- **触发短语要多样**，覆盖正式的、口语的、用户没点名文件类型但明显需要的表达。

## 四、内容安全准则

生成的 skill 不得包含：
- 恶意代码、漏洞利用、旨在未授权访问或数据泄露的内容
- 在 description 说明范围之外给用户带来意外的行为
- 误导性、伪装用途的 skill

角色扮演类需求（「扮演某个角色」）通常没问题，但不得用于社会工程或欺骗。
