# 评测指南

skill 的评测方法论。源自 skill-creator-cn 的评测体系（schemas/grader/comparator），**借鉴思维与契约，不引入 Python 脚本**——重型量化仍复用官方 skill-creator。

## 一、分层验证

评测不是只有「跑 benchmark」一档。默认轻量，进阶可选：

| 档位 | 适用 | 做法 |
|------|------|------|
| **轻量（默认）** | 绝大多数 skill、初学者 | 构造 2–3 个真实 prompt，直接跑一遍，把输出摆给用户看，问「这样对吗？」。零术语、零 JSON |
| **重型量化（进阶）** | 老手、要跨版本对比 | 复用官方 skill-creator：并行 with-skill/baseline 子代理 + benchmark + eval viewer |

**默认永远走轻量**。只有用户主动要求「严格量化对比」或「改完比之前好多少」时才上重型。

## 二、evals 契约 + expectation 写作

### evals.json 最小契约

模板见 `templates/evals/evals.json.template`，字段：
- `skill_name`：与 frontmatter `name` 一致
- `evals[].prompt`：用户真实会说的任务（具体、带场景，别出太明显的题）
- `evals[].expected_output`：期望结果的简要描述（人读）
- `evals[].files`：输入文件（可选）
- `evals[].expectations`：可验证的陈述列表（核心）

### expectation 写作三原则

1. **可验证陈述，而非模糊愿望**。「输出包含客户名和金额」✅；「输出质量高」❌（无法判断）
2. **存疑默认 FAIL**。举证责任在通过方——没有清晰证据就判 FAIL，宁可漏过不可假阳性
3. **有区分度**。好的 expectation：skill 做对时通过、做错时失败。警惕「文件名对但内容空」「碰巧出现关键词」这类**假阳性**——弱 assertion 通过比没有更糟，它制造虚假自信

> **批判评测本身**：每次评完回头看，有没有「做错了也能过」的废 assertion？有没有「做对了却没覆盖」的重要结果？前者删或收紧，后者补上。

## 三、触发评测查询集设计

优化 description 触发率时，手工设计一个查询集（不依赖 run_loop.py 也能用）：

- **20 条查询**，应触发 / 不应触发**各占一半**
- **应触发（8–10 条）**：覆盖正式的、口语的；用户没点名 skill/文件类型但明显需要的情况；本 skill 与其他 skill 竞争但该赢的情况
- **不应触发（8–10 条）**：最有价值的是**近似负例**——和本 skill 共用关键词但实际需要别的东西。别出「写个斐波那契函数」这种太简单的负例，测不出区分度

查询要真实：带文件路径、职位、列名、公司名，有小写/缩写/口语。拿这个集自测 description，看该触发的触发没、不该触发的有没有误触。

## 四、输出质量评估量表

手工评审 skill 输出时，从两个维度打分（各 1–5）：

| 维度 | 1（差） | 3（可） | 5（优） |
|------|---------|---------|---------|
| **Content·正确性** | 重大错误 | 小错 | 完全正确 |
| **Content·完整度** | 缺关键要素 | 基本齐 | 全要素到位 |
| **Content·准确性** | 严重不准 | 轻微不准 | 全程准确 |
| **Structure·组织** | 混乱 | 尚可 | 清晰合逻辑 |
| **Structure·格式** | 不一致/损坏 | 基本一致 | 专业精致 |
| **Structure·可用性** | 难用 | 勉强能用 | 易用 |

按任务调整维度（如 PDF skill 加「字段对齐」「文字可读」）。评分时引用具体证据，别只说「好/差」。

## 五、平台差异与降级

不同运行环境下，评测的降级处理：

| 平台 | 限制 | 降级做法 |
|------|------|----------|
| **Claude Code** | 完整 | 全功能：子代理并发、baseline、viewer、run_loop |
| **Claude.ai / 无子代理** | 无子代理 | 测试用例逐条跑（自己既写又跑，作合理性检查）；跳过 baseline/盲评；以用户定性反馈为准 |
| **headless（Cowork 等）** | 无显示器 | viewer 用 `--static` 输出独立 HTML；用户审阅后从 feedback.json 读反馈 |

简单单步任务（如「读这个 PDF」）即使 description 匹配也可能不触发——Claude 直接用基础工具搞定，这是正常行为，不是 bug，别在评测里强求。
