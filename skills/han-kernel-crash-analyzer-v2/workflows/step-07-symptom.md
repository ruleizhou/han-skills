# Step 7: 区分症状与根因

**在给出最终结论之前，必须做这一步检查：**

```
自问：
- 崩溃位置是受害者还是加害者？
- 如果移除崩溃位置的那行代码，问题会消失还是移到别处？
- 调用栈中最深的函数是崩溃点还是崩溃后果？
```

给出结论时明确标注：
- **直接原因**（触发崩溃的指令/逻辑）
- **根本原因**（导致直接原因的底层缺陷）

## 质量自查清单

在确定最终结论之前，逐项核查：

```
[ ] objdump 是否已执行？（跳过必须有充分理由，在报告中说明）
[ ] fault address 是否已映射到确定的源码行？
[ ] 根因结论是否可证伪？（存在什么样的数据能让此结论不成立？）
[ ] 若涉及 init/probe/校准阶段：回调执行顺序是否已验证？
[ ] 结论是否与 Step 0 收集的场景一致？（不一致是否已标记异常？）
```

**如果任一项不通过：回到对应的步骤补全分析，不得跳过。** 在报告中注明哪些项不适用（如"不涉及 init/probe"）。

**完成后，读取 `workflows/adversarial-verify.md` 的 A 节，启动根因对抗验证。**
