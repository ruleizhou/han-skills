# Debug Step 3: 通用调试模式

## 目标

当没有对应子 skill 时，通过交互式信息收集和五维排查框架帮助用户定位根因。
问题解决后进入反馈闭环，生成分析文档并引导创建子 skill。

## 流程

### 3.1 交互式信息收集（不可跳过）

用 **单次** AskUserQuestion 同时收集问题描述和代码路径：

- **问题 1 — header: "问题描述"**
  - 选项：`[提供问题描述]` — 用户输入问题现象、平台、模块、错误信息、复现步骤等
- **问题 2 — header: "代码路径"**
  - 选项：`[提供代码]` — 用户指定相关源代码路径

### 3.2 上下文读取

收集完成后：
1. 如果用户提供了代码路径 → 用 `Read` 工具读取关键文件，了解技术上下文
2. 检查当前工作目录下的诊断文件：
   ```bash
   ls -lt *.log *.txt *.json *.dmp *.dump 2>/dev/null | head -20
   ```
3. 发现可能的诊断文件后，告知用户，请用户确认哪些相关
4. 如果没有文件，引导用户提供（粘贴日志、指定路径等）

### 3.3 五维排查框架

基于收集的信息，按以下五维展开分析：

**维 1 — 问题细化**：具体表现是什么？（错误信息、异常行为、性能指标）

**维 2 — 信息收集**：什么设备/系统/版本？有没有错误日志？

**维 3 — 定位范围**：
- 哪个组件/子系统出问题？
- 是否和特定硬件/驱动相关？
- 同一环境下其他功能是否正常？
- 出问题的代码区域（结合用户提供的代码路径）

**维 4 — 时间线与变化点**：
- 问题什么时候开始出现？
- 最近有什么变更？（代码更新、配置修改、硬件更换）
- 之前是否正常工作过？回退到哪个版本正常？
- 是否是间歇性问题（时好时坏）？

**维 5 — 收敛与结论**：
1. 列出所有可能的假设（按可能性排序）
2. 对每个假设提出验证方法
3. 给出最可能的根因
4. 提出修复/验证建议

### 3.4 输出分析结论

```
## 通用调试分析

### 问题概况
- 平台：<platform>
- 模块：<module>
- 问题类型：<type>
- 现象：<summary>

### 排查过程
1. <步骤1>
2. <步骤2>
...

### 根因分析
<结论>

### 建议
<修复/验证建议>

### 关键词
<用于后续子 skill 创建的关键词列表>
```

### 3.5 等待用户反馈

分析完成后，**等待用户反馈问题是否解决**。

- 用户发出"解决了/修好了/搞定了/根因确认了/不再复现"信号 → 读取 `workflows/feedback-loop.md`
- 用户发出其他信号 → 继续迭代分析

---

分析输出后等待用户反馈。用户发出"搞定/修好/解决"信号时，读取 `workflows/feedback-loop.md` 进入反馈闭环。
