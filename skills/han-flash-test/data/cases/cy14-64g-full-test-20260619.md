# 测试案例：CY14-64G 顺序/随机性能测试（2026-06-19）

- **日期**: 2026-06-19
- **UFS 型号**: CY14-64G (Samsung 64GB UFS 3.1)
- **Rev**: 0002
- **厂商 ID**: 0x0298 (Samsung)
- **调度器**: none（测试时切换）
- **Write Booster**: ❌ 关闭 (wb_enable=false)

## 测试条件

| 参数 | 值 |
|------|-----|
| 测试方法 | ufs_test.sh (fio 3.33 psync) |
| 读目标 | /dev/block/sda20 (裸 userdata) |
| 写目标 | /dev/block/sda12 (裸 rawdump, 12488 MiB) |
| bs | 512K (顺序) / 4K (随机) |
| size | 1G per job |
| numjobs | 1/2/4/8 (顺序) / 8 (随机) |
| 轮次 | 预热1 + 正式3取中位数 |
| CPU绑核 | 0-5 |

## 规格对标

| 项目 | 规格 | 实测(中位数) | 达标率 | 判定 |
|------|------|-------------|--------|:----:|
| 顺序读 nj=1 | 820 MB/s | 913 MB/s | 111% | ✅ |
| 顺序读 nj=2 | 820 MB/s | 1,402 MB/s | 171% | ✅ |
| 顺序读 nj=4 | 820 MB/s | 1,715 MB/s | 209% | ✅ |
| **顺序读 nj=8** | **820 MB/s** | **1,893 MB/s** | **231%** | **✅** |
| 顺序写 nj=1 | 520 MB/s | 422 MB/s | 81% | ❌ |
| 顺序写 nj=2 | 520 MB/s | 486 MB/s | 93% | ❌ |
| 顺序写 nj=4 | 520 MB/s | 475 MB/s | 91% | ❌ |
| **顺序写 nj=8** | **520 MB/s** | **157 MB/s** | **30%** | **❌** |
| 随机读 | 35000 IOPS | 47,400 IOPS | 135% | ✅ |
| 随机写 | 80000 IOPS | 3,556 IOPS | 4.4% | ⚠️ |

## 关键发现

1. **顺序读全面超越规格** — nj=8 达 **1,893 MB/s**，是规格 820 MB/s 的 231%。读带宽随 numjobs 线性增长，UFS 读通路极健康。比历史测试(1,346 MB/s)大幅提升，得益于 rawdump 裸写释放了读通路全带宽。

2. **顺序写未达标且 SLC 断崖确认** — Write Booster 关闭状态下，峰值仅 486 MB/s(nj=2)，未达 520 MB/s 规格。nj=8(8GB累计)突降至 157 MB/s，为典型 **SLC 缓存耗尽→TLC 直写** 特性，降速 67%。

3. **Write Booster 关闭是写不达标的关键因素** — 该 UFS 支持 WB(wb_type=0x01, wb_max_alloc_units=0x600=1536)但当前系统 `wb_enable=false`。启用 WB 有望恢复乃至超越规格。

4. **随机读达标** — 47,400 IOPS，超规格 35000 IOPS 的 35%。

5. **随机写受限** — 仅 3,556 IOPS，但此为 Android fio psync 引擎(QD1)限制，非 UFS 本身性能。

## 建议

1. **启用 Write Booster**：排查 `wb_enable` 为何为 false，启用后重新测试写性能
2. **大容量写入测试**：已确认 SLC 断崖出现在 ~8GB 累计写入后，TLC 直写约 157 MB/s
3. **后续迭代**：换芯片或复测后说「再看结果」触发反馈闭环
