# 分析手册生成模板

闭环确认 success 后，检查是否需要生成分析手册。生成条件：

- 模式 confidence ≥ 3（已验证有效的成熟模式）
- 或 crash 类型为新类型且用户确认收录
- 或用户主动要求（"生成手册"、"写个分析指导"）

---

## 手册结构

生成的 Markdown 文件写入用户当前工作目录，命名格式：`<CrashType>-分析手册.md`。

手册必须包含以下章节，每一步都要有**可直接复制执行的命令**：

### 一、现象识别（目标：3 分钟内判定 crash 类型）

- 列出用于搜索 crash 签名的 `grep` 命令
- 描述命中/不命中分别意味着什么
- 给出"确认本类型"的判定条件

### 二、DebugImage 深度分析（目标：10 分钟）

- 提取各核 PC 的命令
- 栈分割方法（以 el1_irq 为界）
- 其他核状态速查表（每种状态的含义）
- 关键判断结论模板

### 三、反汇编定位（目标：15 分钟）

- 检查工具的命令（which objdump）
- 从 System.map 取符号地址的命令
- 反汇编命令（带实际地址占位符）
- 如何确认卡死指令和寄存器（注释示例）

### 四、源码因果链追踪（目标：20 分钟）

- 列出所有相关源文件路径
- 用 grep 定位关键函数的命令
- 绘制执行顺序（代码行号标注）
- 验证硬件停止 vs DMA 释放的时序

### 五、修复方案定位（目标：10 分钟）

- 修复原则（一句话）
- 需要修改的位置（文件名 + 上下文）
- Unified diff 模板

### 六、验证方法

- 复现测试方法
- 确认点（dmesg/ramdump 中应不再出现什么）
- 回归测试项

### 附录 A：快速诊断 checklist

- 10 项以内的勾选列表
- 命中阈值（≥ N 项 → 确认）

---

## 生成原则

1. **面向人而不是 AI**：每个步骤都应该是工程师可以独立执行的，不需要 `AskUserQuestion` 交互
2. **命令可复制**：所有 grep/objdump/find 命令使用相对路径占位符（`symbols/vmlinux`、`parser_out/Small/dmesg_TZ.txt`）
3. **地址占位符**：反汇编地址用 `<readl地址>` 等占位符，因为 KASLR 偏移每次不同
4. **代码行号标注**：源码引用标注具体文件和行号
5. **内容从本次案例提取**：反汇编示例、调用栈片段、寄存器名等均从实际分析中提取
