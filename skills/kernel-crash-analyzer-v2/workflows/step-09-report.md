# Step 9: 输出分析报告

用户要求生成报告时，使用以下模板（中文）：

```markdown
# [Crash 类型] 分析报告

## 一、崩溃概览
- 崩溃类型、进程、时间
- 直接原因 vs 根本原因

## 二、证据链
- Fault address → objdump 指令 → 源码位置 → 数据流分析

## 三、调用栈分析
- 逐层分析，标注每层角色

## 四、根因结论
- 明确的根因描述，附带证据

## 五、修复建议
- Unified diff
- [若涉及 init/probe] 执行顺序验证

## 六、验证步骤
- 修复后如何验证效果

## 七、Agent 对抗验证结果（如有）
- **根因验证**: [CONCUR/CONCUR_WITH_CAUTION/DISAGREE] — Agent 主要质疑：<摘要>；主分析者回应：<摘要>
- **修复验证**: [APPROVE/APPROVE_WITH_NOTES/REJECT] — Agent 主要质疑：<摘要>；主分析者回应：<摘要>
- 如 Agent 不可用：标注"Agent 不可用，跳过对抗验证"
```

如果用户只需要分析不需报告，跳过这个模板，直接给出第四、五节的核心内容。

## 案例摘要（嵌入报告，不存档）

在报告末尾附上本次分析的**案例摘要**（以 markdown 形式输出，**不写入文件**）：

```markdown
## 案例摘要（待闭环）
- **id**: <crash_type>
- **crash 类型**: <类型名>
- **fault address**: <地址>
- **平台/内核**: <平台/版本>
- **场景**: <场景>
- **根因分类**: <竞态/错误路径清理/UAF/初始化顺序/锁顺序/其他>
- **根因简述**: <1-2 句>
- **修复摘要**: <修复建议摘要>
- **关键词**: [<kw1>, <kw2>]
- **反汇编工具**: [<工具>]
- **质量自查**: <通过项/未通过项>
- **对抗验证**: [CONCUR+APPROVE | CONCUR_WITH_CAUTION+APPROVE_WITH_NOTES | DISAGREE+REJECT | 跳过（Agent 不可用）]
```

案例文件和模式库的更新**推迟到反馈闭环时统一处理**，不在本轮分析中执行。
