# Step 6: 源码分析

从上一步定位到的源码行出发，追踪因果链。

## 模式匹配

在开始源码追踪之前，检查 `data/patterns.json` 中是否有匹配当前 crash 类型 + 代码区域的已知模式。如果存在高置信度（confidence ≥ 2）模式，将其作为分析线索——但**必须用当前的反汇编证据验证**，模式只是线索不是结论。

## 分析路径

1. **正向追踪**：从崩溃指令回溯数据流，找到导致非法值的源头。
2. **反向验证**：遍历调用栈中每个函数，理解其正常行为和错误路径。
3. **必查清单**（按 crash 类型决定优先级）：
   - **竞态条件**：是否有并发访问同一数据结构但没有适当的锁保护？
   - **锁顺序**：是否存在 mutex A→B 和 mutex B→A 的交叉持有路径？
   - **错误路径清理**：probe/init 失败时的 goto 标签是否正确释放了之前申请的资源？设备是否在半初始化状态被注册到了子系统？
   - **UAF 模式**：释放后指针是否被清零？是否有潜在的后台线程仍在引用？
   - **ERR_PTR 检查**：返回 NULL 的路径是否实际返回了 ERR_PTR，或反之？
   - **初始化顺序**：依赖的资源在 probe 中是否先于使用者初始化？

**完成后，读取 `workflows/step-07-symptom.md` 继续。**
